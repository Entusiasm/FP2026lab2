Вариант 18 – Фабрика игрушек, Tagless Final

Фабрика получает материалы, собирает игрушки и отслеживает брак. Реализация в стиле Tagless Final: бизнес-логика обобщена над F[_]: Monad, эффекты вынесены в алгебры, конкретная монада (IO) подставляется в точке сборки.

Для русских символов в консоли Windows предварительно выполните:

bash
chcp 65001

Запуск
bash
sbt run

Структура проекта
text
src/main/scala/
    ├── algebra/               # 4 TF-алгебры
    │     ├── Console          — ввод/вывод
    │     ├── ConfigAlgebra    — чтение конфигурации (Reader)
    │     ├── Logging          — накопление лога (Writer)
    │     └── StateMachine     — работа с состоянием (State)
    ├── domain/                # типы данных и начальные значения
    │     └── Model.scala      — Product, Material, Recipe, Config, StateData
    ├── interpreters/          # интерпретаторы для IO
    │     ├── ConsoleIO        — реализация Console через IO.print/readLine
    │     ├── ConfigIO         — реализация ConfigAlgebra (конфиг передаётся)
    │     ├── LoggingIO        — реализация Logging (печать в консоль с [LOG])
    │     └── StateMachineIO   — реализация StateMachine (использует Ref[IO, StateData])
    ├── program/               # сценарии меню, обобщённые над F
    │     └── FactoryProgram   — цикл обработки команд, параметризованный F[_]
    └── Main.scala             # точка сборки приложения (IO-интерпретатор)
Ключевая идея
Вся бизнес-логика (FactoryProgram) написана в терминах алгебр и не знает о конкретном эффекте F. Она требует только Monad[F] и экземпляры Console[F], ConfigAlgebra[F], Logging[F], StateMachine[F]. Конкретные интерпретаторы для IO подключаются в Main.

Что изменилось по сравнению с исходной лабораторной (обычные монады)
Отказ от «голых» монад Reader, Writer, State – вместо них введены алгебры:

Reader → ConfigAlgebra (чтение рецептов, цен, правил)

Writer → Logging (накопление и вывод логов)

State → StateMachine (получение/изменение состояния, бизнес-операции)

Алгебра Console выделена из IO – теперь ввод/вывод не вшит в эффект, а является независимым требованием.

Программа параметризована типом F[_] – можно подставить IO, Id, Future, ZIO и т.д.

Состояние больше не передаётся неявно через State; вместо этого используется Ref[IO, StateData] в интерпретаторе StateMachineIO. Это соответствует идиомам cats-effect.

Логирование больше не склеивается автоматически монадой Writer; алгебра Logging требует явного вызова log в нужных местах, но интерпретатор может накапливать строки или сразу печатать (в нашем случае печать с префиксом [LOG]).

Конфигурация теперь не читается из Reader, а передаётся как обычный параметр в конструктор интерпретатора ConfigIO. В алгебре ConfigAlgebra методы возвращают F[?], но внутри интерпретатора для IO они просто IO.pure(config.recipes(...)).


Что осталось от предыдущей версии
Предметная область (Product, Material, Recipe, Config, StateData) полностью сохранена.

Логика производства (списание материалов, расчёт брака, обновление готовых изделий) осталась без изменений – она просто обёрнута в методы StateMachine.

Логика чтения команд и отображения меню структурно та же, но переписана на for-comprehension с вызовами алгебр.
