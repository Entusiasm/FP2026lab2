# Вариант 18 – Фабрика игрушек, Tagless Final

Фабрика получает материалы, собирает игрушки и отслеживает брак. Реализация в стиле **Tagless Final**: бизнес-логика обобщена над `F[_]: Monad`, эффекты вынесены в алгебры, конкретная монада (`IO`) подставляется в точке сборки. Состояние передаётся **явно** (без `Ref`), меню построено композиционно (`MenuTreeNode` / `MenuLeaf`).

## Запуск

Для русских символов в консоли Windows предварительно выполните:

```bash
chcp 65001
Затем:

bash
sbt run
Управление
Вводите номер пункта меню и нажмите Enter.

Структура проекта
text
src/main/scala/
    ├── algebra/               # 4 TF-алгебры
    │     ├── Console.scala          — ввод/вывод
    │     ├── ConfigAlgebra.scala    — чтение конфигурации (Reader)
    │     ├── Logging.scala          — накопление лога (Writer)
    │     └── StateMachine.scala     — работа с состоянием (State)
    ├── domain/                # типы данных и переходы состояния
    │     ├── Model.scala            — Product, Material, Recipe, Config, StateData
    │     └── StateTransitions.scala — чистые функции изменения состояния
    ├── interpreters/          # интерпретаторы для IO
    │     ├── ConsoleIO.scala        — реализация Console через IO.print/readLine
    │     ├── ConfigIO.scala         — реализация ConfigAlgebra (конфиг передаётся)
    │     ├── LoggingIO.scala        — реализация Logging (печать в консоль с [LOG])
    │     └── StateMachineIO.scala   — реализация StateMachine (явное состояние, без Ref)
    ├── ui/                     # композиционное меню
    │     ├── MenuOption.scala       — MenuLeaf / MenuTreeNode
    │     ├── MenuActions.scala      — конкретные действия для каждого пункта
    │     └── MainMenu.scala         — запуск меню
    └── Main.scala              # точка сборки приложения (IO-интерпретатор)
Ключевая идея
Вся бизнес-логика (MenuActions) написана в терминах алгебр и не знает о конкретном эффекте F. Она требует только Monad[F] и экземпляры Console[F], ConfigAlgebra[F], Logging[F], StateMachine[F]. Конкретные интерпретаторы для IO подключаются в Main.scala. Состояние передаётся явно через параметры методов StateMachine и возвращается как часть результата.

Что изменилось по сравнению с исходной лабораторной (обычные монады)
Отказ от «голых» монад Reader, Writer, State – вместо них введены алгебры:

Reader → ConfigAlgebra (чтение рецептов, цен, правил)

Writer → Logging (накопление и вывод логов)

State → StateMachine (получение/изменение состояния, бизнес-операции)

Алгебра Console выделена из IO – ввод/вывод стал независимым требованием.

Программа параметризована типом F[_] – можно подставить IO, Id, Future, ZIO и т.д.

Состояние больше не прячется в Ref – все методы StateMachine принимают StateData и возвращают новое состояние вместе с результатом (F[(StateData, A)]). Это позволяет обойтись без Ref и сохранить чистоту.

Логирование больше не склеивается автоматически монадой Writer; алгебра Logging требует явного вызова log, но интерпретатор (LoggingIO) выводит логи на консоль с префиксом [LOG].

Конфигурация передаётся как обычный параметр в конструктор интерпретатора ConfigIO, методы алгебры просто возвращают IO.pure(config.recipes(...)).

Меню стало композиционным – больше нет огромного match. Каждый пункт – отдельный MenuLeaf, главное меню – MenuTreeNode. Добавление нового пункта требует лишь дописать строку в Seq и определить соответствующую функцию действия.

Что осталось от предыдущей версии
Предметная область (Product, Material, Recipe, Config, StateData) полностью сохранена.

Логика производства (списание материалов, расчёт брака, обновление готовых изделий) осталась без изменений – она обёрнута в чистые функции в domain.StateTransitions.

Логика чтения команд и отображения меню структурно та же, но переписана на вызовы алгебр.
